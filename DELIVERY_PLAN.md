# Pearls Migrator — Delivery Plan

Этот файл фиксирует порядок ближайших спринтов, релизов и коммитов.

`ARCHITECTURE.md` описывает целевую архитектуру. `DELIVERY_PLAN.md` описывает, как идти к ней маленькими проверяемыми шагами.

## Рабочий Принцип

- Source PDFs в `pearls/` — сырой архив от редакторов.
- Reviewed JSON в `data/parsed/{year}/` — источник правды для приложения.
- `DOCUMENTS_GUIDE.md` — источник правил по типам документов, трем датам, заголовку, телу и футеру.
- Postgres — runtime index, который всегда можно пересобрать из JSON.
- MVP режется до минимально работающего публичного каталога.
- Все необязательные фичи, которые тормозят релиз, переносятся после деплоя.
- Каждый этап закрывается отдельным коммитом.
- После завершения этапа отмечаем чекбоксы в этом файле.

## MVP Scope

Первый публичный релиз должен делать только это:

- показать общий список материалов;
- сгруппировать материалы по году и месяцу публикации на сайте;
- сортировать новые материалы выше старых;
- дать фильтр только по году публикации на сайте;
- открыть страницу материала для чтения;
- для составных PDF показать на карточке выпуска найденные внутренние диктовки/лекции: автор + название, насколько это надежно извлекается;
- генерировать индивидуальные TXT/DOCX/EPUB скачивания отдельной deploy-time командой;
- использовать Postgres как runtime index, пересобираемый из reviewed JSON;
- иметь нормальный дизайн, достаточный для публичного запуска;
- задеплоиться в production.

Не входит в MVP:

- фильтр по автору;
- страницы авторов;
- фильтр по историческому году создания;
- фильтр по типу документа;
- полноценное разбиение составного PDF на отдельные публичные страницы;
- общий архив всех документов;
- bulk ZIP downloads всех документов;
- RAG, чат, embeddings, Qdrant;
- админка или модераторская панель.

### Sprint 2.1 — Надежная Дата Сайта И Фильтр По Году

Цель: перед первым публичным релизом оставить в интерфейсе только надежную навигацию по дате публикации на сайте.

Причина: старые файлы часто имеют имена без даты (`Death_2.pdf`, `Be gone, forces of anti-Love.pdf`), составные PDF могут смешивать авторов и футеры, а месяц публикации сайта часто нужно брать из начала текста.

- [ ] Считать год публикации сайта из папки `pearls/{year}/` / `data/parsed/{year}/`
- [ ] Считать месяц публикации сайта из первых строк документа с OCR-нормализацией (`202 6` → `2026`)
- [ ] Использовать квартальное имя файла только как дополнительный источник для новых файлов
- [ ] Проверить `sitePublication` на старых файлах с описательными именами
- [ ] Временно убрать из UI фильтры по автору, году создания и типу документа
- [ ] Оставить публичный фильтр только по году публикации на сайте
- [ ] Оставить группировку списка по году и месяцу публикации на сайте
- [ ] Пересидить Postgres из исправленных JSON

План коммитов:

1. `Уточнить дату публикации сайта`

### Sprint 2.2 — Составные Документы В MVP

Цель: не дробить составный PDF на отдельные страницы, но показать в списке, какие внутренние материалы в нем есть.

Минимальная модель для MVP:

- один PDF остается одной карточкой и одной страницей чтения;
- внутри JSON появляется или вычисляется список внутренних материалов для отображения;
- для каждого внутреннего материала пытаемся показать `author`, `title`, `rawHeader`;
- если автор или название не уверены, показываем безопасную исходную строку, а не выдуманное значение;
- эти данные используются только для отображения в карточке/странице, не для фильтров.

Маркерный подход:

- искать длинный разделитель футера (`_____`, `-----` и похожие);
- после футера ожидать новый блок коротких строк-заголовков;
- искать строки вида `Диктовка ...`, `Лекция ...`, `Проповедь ...`;
- строку `Том ... № ...` использовать как слабый дополнительный сигнал, но не как обязательный критерий;
- не подключать модель на этом этапе.

- [ ] Найти реальные составные PDF в `data/parsed/` и выбрать 5-10 контрольных примеров
- [ ] Добавить минимальное извлечение внутренних заголовков/авторов для карточки выпуска
- [ ] Обновить типы JSON/DB или runtime DTO под список `containedDocuments`
- [ ] Вывести внутренние заголовки на карточке каталога
- [ ] Вывести внутренние заголовки на странице материала
- [ ] Проверить `2026Q1-3`, `2025Q1-3` и старые файлы с описательными именами
- [ ] Пересидить Postgres из исправленных JSON

План коммитов:

1. `Найти заголовки внутри составных выпусков`
2. `Показать состав выпусков в каталоге`

## Quarterly Content Release Flow

Повторяемый флоу для новых пакетов лекций:

1. Получить новые PDF от редакторов.
2. Положить PDF в `pearls/{year}/`.
3. Запустить локальный парсинг.
4. Проверить JSON в `data/parsed/{year}/`.
5. Если JSON корректный, сделать коммит.
6. Запушить в `main`.
7. Задеплоить через PM2.
8. Запустить seed на production Postgres.
9. Проверить каталог по годам и чтение материалов.

План коммита для контентного релиза:

`Добавить лекции за YYYY QN`
