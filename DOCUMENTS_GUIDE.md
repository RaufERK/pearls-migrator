# Pearls Migrator — Documents Guide

Этот файл фиксирует смысловую модель документов. Он нужен, чтобы парсер, JSON, база данных, каталог и будущая модель обработки текста одинаково понимали, где заголовок, где тело, где футер и какие даты у документа существуют.

## Главная Идея

Физический файл-источник не является просто лекцией с одним заголовком и одной датой. На текущем этапе основным источником считается Word-брошюра из `data/source-data/pearls-word/`. В одной брошюре могут быть:

- редакционная дата публикации на нашем сайте;
- историческая дата создания документа;
- дата публикации в серии `Жемчужины Мудрости`;
- тип документа: диктовка, лекция, проповедь, призыв, материал;
- несколько строк заголовка;
- тело документа;
- финальная сноска после черты с историческим контекстом.

Reviewed JSON в `data/parsed/` должен сохранять эти данные явно, даже если часть полей пока не используется на сайте. Эти JSON-файлы нельзя чинить руками: они должны меняться только через парсер, нормализацию метаданных и AI-пайплайн.

## Правило Для `data/parsed`

`data/parsed/` — это результат работы программы, а не место для ручной редакторской правки. Если в JSON неверно определены автор, тип документа, название, дата, футер, заголовок или разбиение на внутренние документы, нужно исправлять:

- логику парсинга Word-брошюр;
- нормализацию метаданных;
- схему и промпт модели в `src/metadataAi.ts`;
- CLI-пайплайн, который прогоняет документы.

После этого нужно заново прогнать нужный диапазон документов и смотреть, воспроизводится ли улучшение автоматически. Ручные правки в `data/parsed/` запрещены, потому что они скрывают ошибки парсера, делают проверку нерепрезентативной и всё равно будут затёрты следующим проходом пайплайна.

Редакторские исключения нужно хранить в `data/word-processing-map.json`: проверенные названия, ожидаемое количество внутренних документов и split markers. Это не ручная правка generated JSON, а часть воспроизводимого Word pipeline.

## Термины: Жемчужина И Внутренний Документ

`Жемчужина` или `файл-документ` — это одна месячная Word-брошюра из `data/source-data/pearls-word/`, которая публикуется на нашем сайте как выпуск. Для жемчужины хранятся только контейнерные данные: путь к исходному Word-файлу, путь к подготовленному DOCX из `data/word-docx/`, путь к JSON, дата публикации на нашем сайте, техническая информация парсинга и общий заголовок серии.

`Внутренний документ` — это отдельная лекция, диктовка, проповедь, молитвенный блок или другой материал внутри жемчужины. В одной жемчужине обычно 1 материал, но модель должна поддерживать до 4 материалов. Бывает и обратная ситуация: один длинный внутренний документ может продолжаться в нескольких жемчужинах разных месяцев. На этапе JSON это фиксируется через собственные `documents[]` в каждом файле, без базы данных и без склейки материалов между файлами.

Все смысловые метаданные — автор, тип, название, дата создания, публикация в серии `Жемчужины Мудрости`, сырой заголовок, тело и футер — относятся к внутреннему документу, а не к жемчужине-контейнеру.

`dictation` означает эзотерическое послание Вознесенного Владыки. `lecture` означает лекцию Посланника. Для архива оба типа близки по способу хранения, но тип нужно сохранять отдельно, чтобы позже строить фильтры и страницы авторов/владык.

В каталоге `простая карточка` — это карточка жемчужины с одним внутренним документом. `Составная карточка` — карточка жемчужины с двумя и более внутренними документами. В обоих случаях порядок отображения внутреннего документа одинаковый: тип, автор, название, дата создания в скобках.

## Важный Поворот: Брошюра Может Быть Контейнером

Одна Word-брошюра не всегда равна одному смысловому документу. В одной брошюре могут быть 1-4 самостоятельных материала: лекции, диктовки, проповеди или обращения. У каждого такого материала могут быть собственные:

- заголовок;
- строка `Том ... № ...`;
- автор;
- тип документа;
- историческая дата создания;
- тело;
- футер после разделителя.

Поэтому нельзя брать первый заголовок брошюры и последний футер брошюры как единую пару. Это приводит к смешиванию метаданных: автор может быть взят из первого материала, а дата, тип или контекст — из футера последнего материала.

Правильная смысловая модель:

- Word-брошюра — это файл-источник или выпуск-контейнер;
- внутри брошюры находится список `documents` / `items`;
- каждый внутренний материал имеет собственные `header`, `body`, `footer` и собственные метаданные;
- метаданные материала можно извлекать только из его собственного заголовка, тела около начала и собственного футера;
- общая дата публикации на сайте может принадлежать контейнеру и наследоваться внутренними материалами, если внутри материала не указана более точная редакционная дата.

Пример целевой идеи:

```json
{
  "sourceWord": "data/source-data/pearls-word/2022/1-й квартал/Брошюры/ЖМ 1_кв. 1_январь 2022_....doc",
  "preparedDocx": "data/word-docx/2022/1-й квартал/Брошюры/ЖМ 1_кв. 1_январь 2022_....docx",
  "sitePublication": {
    "label": "Январь 2022",
    "rawLabel": "Январь 2022",
    "year": 2022,
    "month": 1,
    "months": ["2022-01"],
    "sortDate": "2022-01-01"
  },
  "documents": [
    {
      "documentTitle": "Первый материал",
      "documentType": "dictation",
      "author": {
        "name": "Господь Майтрейя",
        "slug": "lord-maitreya",
        "raw": "Диктовка Господа Майтрейи"
      },
      "parts": {
        "header": [],
        "body": [],
        "footer": []
      }
    },
    {
      "documentTitle": "Второй материал",
      "documentType": "dictation",
      "author": {
        "name": "Ланелло",
        "slug": "lanello",
        "raw": "Том 56, № 5 – возлюбленный Ланелло – 1 марта 2013 г."
      },
      "parts": {
        "header": [],
        "body": [],
        "footer": []
      }
    }
  ]
}
```

Верхнеуровневые поля лекции (`documentTitle`, `documentType`, `author`, `parts`) больше не используются в чистой модели. Они должны находиться только внутри `documents[]`.

## Типы Документов

Основные типы:

- `dictation` — диктовка;
- `lecture` — лекция;
- `lectureCourse` — курс лекций;
- `teaching` — учения;
- `sermon` — проповедь;
- `prayer` — призыв или молитвенный блок, если документ фактически начинается с призыва;
- `material` — другой материал, если тип нельзя надежно определить.

Тип обычно определяется по заголовкам и футеру:

- `Диктовка Эль Мории` → `dictation`;
- `Лекция Элизабет Клэр Профет` → `lecture`;
- `Курс лекций ...` → `lectureCourse`;
- `Учения ...` → `teaching`;
- `Проповедь ...` → `sermon`;
- `Открывающий призыв`, `Призыв`, `Призыв Посланника` → начало тела, а не продолжение заголовка и не новый внутренний документ.

Если тип найден в футере точнее, чем в заголовке, футер считается важным источником для уточнения метаданных.

## Три Даты Документа

### 1. Дата публикации на нашем сайте

Это редакционная дата, по которой документ входит в календарь сайта. Для нового Word-flow главным источником этой даты является путь и имя исходной Word-брошюры в `data/source-data/pearls-word/`: год берется из папки года, квартал — из папки квартала, месяц — из имени файла. Текст документа и колонтитулы используются как подтверждающие или дополнительные источники, а не как первый источник правды.

В Word-брошюрах дата также может находиться в теле документа, верхнем или нижнем колонтитуле. Парсер обязан читать `word/header*.xml` и `word/footer*.xml` из DOCX, но если при конвертации `.doc -> .docx` колонтитулы потерялись, `sitePublication` всё равно должна восстанавливаться из исходного пути и имени файла.

Примеры строк:

```text
Октябрь 2006
Ноябрь 2006
Март 2006
```

Эта дата нужна для основного каталога сайта: список документов по годам и месяцам публикации у нас.

Один большой документ может относиться сразу к нескольким месяцам. Поэтому в целевой модели нужно хранить не только одну дату, но и список месяцев публикации.

Перед распознаванием даты нужно делать OCR-нормализацию технических разрывов внутри чисел. Например, `Май 202 6` и `Февраль 202 6` должны распознаваться как `Май 2026` и `Февраль 2026`. При этом исходную строку нельзя терять: `rawLabel` хранит текст как в PDF, а `label` хранит нормализованную человекочитаемую форму.

Рекомендуемые поля:

```json
{
  "sitePublication": {
    "label": "Октябрь 2006",
    "rawLabel": "Октябрь 2006",
    "year": 2006,
    "month": 10,
    "months": ["2006-10"],
    "sortDate": "2006-10-01"
  }
}
```

Если документ покрывает несколько месяцев:

```json
{
  "sitePublication": {
    "label": "Октябрь - Ноябрь 2006",
    "rawLabel": "Октябрь - Ноябрь 2006",
    "year": 2006,
    "month": 10,
    "months": ["2006-10", "2006-11"],
    "sortDate": "2006-10-01"
  }
}
```

### 2. Дата создания или рождения документа

Это дата, когда диктовка была реально продиктована, лекция дана, проповедь произнесена или материал впервые создан.

Чаще всего она находится в футере после черты:

```text
____________________________________________________
Лекция Элизабет Клэр Профет ... была дана в понедельник, 4 января 1988 года, в Ройял Тетон Рэнч, Монтана.
```

Эта дата нужна для исторического контекста. Пользователь должен иметь возможность нажать на год рождения документа и увидеть все документы, созданные в этом году.

Рекомендуемые поля:

```json
{
  "creation": {
    "date": "1988-01-04",
    "year": 1988,
    "raw": "Лекция Элизабет Клэр Профет ... была дана в понедельник, 4 января 1988 года, в Ройял Тетон Рэнч, Монтана."
  }
}
```

Если точная дата не извлечена, но год понятен, сохраняем год и исходный текст:

```json
{
  "creation": {
    "date": null,
    "year": 1991,
    "raw": "Диктовка Сен-Жермена была дана ... в воскресенье, 11 августа 1991 года ..."
  }
}
```

### 3. Дата публикации в `Жемчужинах Мудрости`

Это дата публикации головной организацией в виде выпуска `Жемчужин Мудрости`. Обычно строка содержит том, номер, автора и дату:

```text
Том 43 №№ 49, 50 – Элизабет Клэр Профет – 3, 10 декабря 2000 г.
Том 35 № 58 – возлюбленный Архангел Чамуил – 29 ноября 1992 г.
```

Эта информация есть не везде, но если она найдена, ее нужно сохранять отдельно.

Рекомендуемые поля:

```json
{
  "pearlPublication": {
    "volume": 43,
    "issue": "49, 50",
    "date": "2000-12-03",
    "rawDate": "3, 10 декабря 2000 г.",
    "raw": "Том 43 №№ 49, 50 – Элизабет Клэр Профет – 3, 10 декабря 2000 г."
  }
}
```

Если в строке указано несколько дат, `date` хранит первую дату для сортировки, а `rawDate` сохраняет исходное значение полностью.

### Практический смысл трех дат

У одного внутреннего документа могут быть три разные даты, и они не должны подменять друг друга:

1. `creation` — самая ранняя историческая дата: когда диктовка была дана, лекция прочитана, проповедь произнесена или материал создан. Обычно находится только в футере: `Диктовка Господа Майтрейи была передана через Э. К. Профет 24 марта 1985 года...`.
2. `pearlPublication` — дата публикации в англоязычной серии `Жемчужины Мудрости` / The Summit Lighthouse. Обычно находится в строке `Том ... № ... – автор – дата`, например `Том 28, № 53 – Господь Майтрейя – 30 декабря 1985 г.`.
3. `sitePublication` — дата русской публикации на нашем сайте, например `Июнь 2026`.

Хронологически обычно идет так: сначала `creation`, затем `pearlPublication`, затем `sitePublication`. Для исторического года документа нужно использовать `creation.year`, а не дату из строки `Том ... № ...`. Дата из строки `Том ... № ...` сохраняется отдельно в `pearlPublication` и может отображаться как информация о выпуске `Жемчужин Мудрости`.

## Автор

Автор должен быть отдельным нормализованным полем. Пользователь должен иметь возможность нажать на автора и увидеть только документы этого автора.

Источник автора:

- заголовок: `Диктовка Эль Мории`, `Лекция Элизабет Клэр Профет`;
- строка `Том ... – автор – дата`;
- футер после черты;
- имя файла — только запасной источник, если текст не дал автора.

Нормализация автора:

- `Э. К. Профет`, `Э.К. Профет`, `Е. К. Профет`, `Возлюбленный Посланник`, `Посланник`, если по контексту речь о ней, нормализуются в `Элизабет Клэр Профет`;
- имя автора всегда должно начинаться с большой буквы: `возлюбленная Роза Света` → `Возлюбленная Роза Света`, `возлюбленная Гуань Инь` → `Возлюбленная Гуань Инь`;
- `raw` сохраняет исходную строку из PDF без потери OCR-особенностей, а `name` хранит нормализованную человекочитаемую форму.

Рекомендуемые поля:

```json
{
  "author": {
    "name": "Эль Мория",
    "slug": "el-morya",
    "raw": "Диктовка Эль Мории"
  }
}
```

Текущее поле `speaker` можно сохранить на переходный период, но целевая модель должна использовать `author`.

## Заголовок, Тело И Футер

Документ нужно хранить как три смысловые части, где тело находится только в `parts.body`.

### Заголовок

Заголовок включает строки до начала тела:

- редакционный месяц сайта;
- тип и автора;
- название документа;
- дополнительные строки заголовка.

Но если строка означает начало самого текста, она уже не должна считаться заголовком.

Маркеры начала тела:

- `Открывающий призыв`;
- `Призыв`;
- `Молитва`;
- `Преамбула`;
- другие похожие строки, которые фактически начинают текст документа.

Такие строки нужно сохранить в теле, но можно дополнительно пометить как первый раздел тела.

Эти маркеры не должны становиться `documentTitle`. Например, если после заголовка идет строка `ПРИЗЫВ`, это означает начало тела документа, а не название лекции или диктовки.

Название документа нужно очищать от дублирования автора, если автор уже сохранен в `author.name`. Например:

- `Учения Элизабет Клэр Профет по Книге Откровения` → `Учения по Книге Откровения`;
- `Проповедь Э. К. Профет по Книге Исаии и Евангелию от Марка` при авторе `Элизабет Клэр Профет` можно отображать как `Проповедь по Книге Исаии и Евангелию от Марка`;
- если заголовок выглядит как `Проповедь ...`, `Лекция ...` или `Диктовка ...`, но отдельного короткого названия нет, допускается использовать эту строку как `documentTitle`, предварительно убрав повтор автора.

В Word pipeline заголовок определяется не только по тексту, но и по форматированию DOCX. Сильные сигналы: жирный текст, увеличенный размер шрифта, курсив в строке названия и отдельный style id заголовочного абзаца. Форматирование не должно побеждать смысловые запреты: служебные строки `Жемчужины Мудрости`, `Том ...`, `ПРИЗЫВ`, автор-only строки, даты выпуска и retreat metadata не становятся `documentTitle`, даже если они выделены жирным или крупным шрифтом.

### Тело

Тело — основной текст документа без заголовочных строк и без финальной сноски после черты.

Для чтения на сайте, скачиваний, поиска и RAG основным содержимым считается именно тело.

### Футер

Футер — финальная информация после длинной черты из `_`, `-` или похожего визуального разделителя.

Футер нельзя выбрасывать. В нем часто находится:

- историческая дата создания;
- автор;
- место события;
- точная формулировка типа документа;
- контекст конференции или события.

Футер нужно хранить отдельно и не смешивать с основным телом.

## Целевая JSON-Модель

Минимальная целевая форма жемчужины-контейнера:

```json
{
  "slug": "1988-01-04-mother-mb2",
  "title": "Жемчужины Мудрости",
  "sitePublication": {
    "label": "Ноябрь 2006",
    "rawLabel": "Ноябрь 2006",
    "year": 2006,
    "month": 11,
    "months": ["2006-11"],
    "sortDate": "2006-11-01"
  },
  "documentsCount": 1,
  "documents": [
    {
      "documentTitle": "Благотворное воздействие макробиотического питания на здоровье",
      "documentType": "lecture",
      "author": {
        "name": "Элизабет Клэр Профет",
        "slug": "elizabeth-clare-prophet",
        "raw": "Лекция Элизабет Клэр Профет"
      },
      "creation": {
        "date": "1988-01-04",
        "year": 1988,
        "raw": "Лекция Элизабет Клэр Профет ... была дана в понедельник, 4 января 1988 года, в Ройял Тетон Рэнч, Монтана."
      },
      "pearlPublication": {
        "volume": null,
        "issue": null,
```

## Правила Извлечения

Парсер должен идти в таком порядке:

1. Извлечь все строки и абзацы из Word-брошюры.
2. Сохранить исходный текст строк, признаки форматирования DOCX и отдельно подготовить нормализованную копию для анализа.
3. В нормализованной копии исправить технические OCR-разрывы внутри чисел и дат: `202 6` → `2026`, `2 9 декабря` → `29 декабря`.
4. Найти редакционную дату сайта в первых строках контейнера.
5. Найти границы внутренних материалов. Сильные маркеры новой части: новая строка `Том ... № ...`, новый явный заголовок типа `Диктовка ...`, `Лекция ...`, `Проповедь ...`, особенно если перед этим уже был футер после разделителя.
6. Для каждого внутреннего материала отдельно найти строки заголовка до первого явного маркера тела.
7. Для каждого внутреннего материала отдельно найти его футер после ближайшего разделителя.
8. Из заголовка и футера этого же материала определить `documentType`, `author`, `documentTitle`.
9. Из футера этого же материала извлечь `creation`.
10. Из строки `Том ... № ...` этого же материала извлечь `pearlPublication`, если она есть.
11. Применить `data/word-processing-map.json`, если для источника есть редакторски проверенные названия или правила разбиения.
12. Сохранить исходные строки каждого материала в `documents[].parts.header`, `documents[].parts.body`, `documents[].parts.footer`.

Если парсер не уверен, он должен сохранить `raw`-значение и поставить нормализованное поле в `null`, а не терять информацию.

Модель можно подключать как вспомогательный этап, но не как первый и единственный источник правды. Сначала нужно детерминированно выделить кандидаты на материалы, заголовки и футеры правилами Word-парсера. Модели можно отдавать уже подготовленный сегмент или короткий набор кандидатов, чтобы она помогала разрешить неоднозначность: где настоящий заголовок, кто автор, какая строка является названием. Давать модели всю брошюру целиком рискованно: дороже, менее воспроизводимо и проще получить смешанные метаданные из разных материалов.

Практический AI-этап для исторического массива:

1. Детерминированный парсер сначала определяет `documentsCount` и границы каждого внутреннего документа.
2. Для каждого внутреннего документа парсер отдельно выделяет `parts.header` и `parts.footer`; `parts.body` модели не отправляется, если не нужен короткий начальный контекст для названия.
3. Модели передается только компактный JSON-кандидат: `sourceWord` или `sourcePath`, `sitePublication`, `header`, `footer`, первые 1-3 строки тела при необходимости и текущие найденные значения.
4. Модель возвращает только строгий JSON с `documentTitle`, `documentType`, `author`, `creation`, `pearlPublication` и признаком уверенности по спорным полям.
5. Результат модели проходит локальную нормализацию: канонические имена авторов, капитализация имени автора, ISO-даты, проверка года и сохранение всех исходных строк в `raw`.
6. Если ответ модели невалидный или противоречит исходным строкам, JSON не переписывается автоматически: поле остается `null`, а исходные `raw`-данные сохраняются для ручной проверки.

## Каталог И Фильтры

Основной каталог сайта строится по `sitePublication.year` и `sitePublication.months`. Для первого публичного релиза надежными считаются только год и месяц публикации на сайте.

Год публикации сайта берется из папки `data/source-data/pearls-word/{year}/`. Квартал берется из папки квартала. Месяц публикации сайта берется из имени исходной Word-брошюры (`январь`, `февраль`, `март` и т.д.). Колонтитулы DOCX и первые строки документа используются только если путь или имя файла неоднозначны либо для проверки качества извлечения.

Для составной брошюры будущие фильтры должны работать по внутренним материалам, а не только по контейнеру. Если контейнер опубликован в 2022 году и содержит несколько материалов, все внутренние материалы должны попадать в фильтр 2022 года, если они наследуют эту дату публикации сайта.

MVP-каталог:

- нажать на год публикации сайта → список материалов за этот год;
- месяцы используются для группировки и сортировки внутри года;
- публичный фильтр по месяцу пока не нужен.

Для составной брошюры в MVP достаточно показать в карточке выпуска внутренний состав: автор и название найденных диктовок/лекций, либо исходную строку заголовка, если уверенно нормализовать автора/название нельзя. Это отображение не должно включать фильтры, отдельные URL или отдельные страницы внутренних материалов до следующего этапа.

Будущие пользовательские переходы после стабилизации составных документов и справочника авторов:

- нажать на автора → список документов этого автора;
- нажать на год создания → список документов, созданных в этом году;
- нажать на тип документа → список диктовок, лекций, проповедей или материалов.

База данных является runtime index. JSON остается источником правды. Если данные в базе и JSON различаются, побеждает JSON, а базу нужно пересидить.

## Практический Приоритет

Ближайшая задача — не идеально распарсить все исторические случаи, а перестать терять важные части документа.

Приоритет полей:

1. Сохранять `parts.header`, `parts.body`, `parts.footer` для одиночного материала и `documents[].parts.header`, `documents[].parts.body`, `documents[].parts.footer` для составной брошюры.
2. Явно хранить `sitePublication`.
3. Явно хранить `creation.raw` и `creation.year/date`, если удалось.
4. Явно хранить `author`.
5. Извлекать `pearlPublication`, если есть строка `Том ... № ...`.
6. Нормализовать `documentType`.
