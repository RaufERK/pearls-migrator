# Миграция Концепции: Word-Брошюры Вместо PDF

Этот документ фиксирует новую рабочую концепцию архива. Он не удаляет и не заменяет старую модель из `DOCUMENTS_GUIDE.md`, а описывает следующий смысловой поворот перед будущей переработкой парсера.

## Подтвержденное Решение

С этого этапа основной источник данных — русские Word-брошюры из `data/source-data/pearls-word/`.

PDF больше не парсим как основной источник: это оказалось сложнее и менее надежно для текущего MVP. Архив PDF хранится внутри `data/source-data/pearls-pdf/` и может позже использоваться только как справочная ссылка на исходный файл.

Первый рабочий диапазон был 2022 год:

```text
data/source-data/pearls-word/2022/1-й квартал/Брошюры
data/source-data/pearls-word/2022/2-й квартал/БРОШЮРЫ
data/source-data/pearls-word/2022/3-й квартал/БРОШЮРЫ
data/source-data/pearls-word/2022/4-й квартал/Брошюры
```

В этих папках исходники в формате `.doc`, поэтому первый технический шаг pipeline — автоматическая конвертация `.doc` в `.docx` через установленный LibreOffice. Подготовленные `.docx` сохраняются отдельно в `data/word-docx/`. Только после этого файл отдается Word-парсеру.

Статус на 5 июня 2026: этот диапазон и все найденные текущие Word-брошюры до `2026 Q2` уже подготовлены, распарсены, проверены по основным проблемным заголовкам, пересидированы в Postgres и пересобраны в downloads.

## Главная Идея

Физическим источником контента становятся квартальные наборы Word-брошюр. Одна брошюра соответствует одному месячному выпуску `Жемчужин Мудрости` на сайте.

PDF больше не считается основным источником для парсинга. Папка `data/source-data/pearls-pdf/` остается архивом исходных PDF-файлов и может использоваться позже только как справочная ссылка на оригинальный файл, если такая связь нужна на сайте.

Новая модель сохраняет идею `Жемчужины` как физического контейнера: внутри одной месячной брошюры может быть один или несколько материалов. Отличие от старого подхода только в источнике: вместо PDF парсим более чистый Word-файл.

Отдельные файлы из папок `Оригиналы` остаются справочным и редакторским источником, но не являются главным входом для MVP-парсинга.

## Источник Данных

Основная папка источников:

```text
data/source-data/pearls-word/
```

Архив старых PDF:

```text
data/source-data/pearls-pdf/
```

Подготовленные Word-файлы после конвертации:

```text
data/word-docx/
```

Целевая организация источников:

```text
data/source-data/pearls-word/
  2025/
    1-й квартал/
      Брошюры/
        ЖМ 1_кв.1_январь 2025_....docx
        ЖМ 2_кв.1_февраль 2025_....docx
        ЖМ 3_кв.1_март 2025_....docx
      Оригиналы/
        ...
```

Главная папка для парсинга:

```text
data/source-data/pearls-word/<год>/<квартал>/Брошюры/
```

Название папки с брошюрами нужно искать без учета регистра, потому что в архиве встречаются оба варианта: `Брошюры` и `БРОШЮРЫ`.

В одном квартале ожидается три месячных брошюры. Подготовка и парсинг должны уметь идти по всем найденным годам и кварталам, но проверять результат лучше квартал за кварталом.

## Что Парсим

Парсер должен обрабатывать русские Word-брошюры из папок `Брошюры`.

Рабочий формат парсера — `.docx`.

Если исходник лежит в старом формате `.doc`, его нужно автоматически конвертировать в `.docx` перед парсингом. Сам парсер не должен пытаться напрямую читать `.doc`, потому что это старый бинарный формат Word и он менее стабилен для извлечения текста.

На первом этапе не используем как основной источник:

- PDF-файлы из `data/source-data/pearls-pdf`;
- отдельные файлы из `Оригиналы`;
- временные Word-файлы вида `~$...`;
- английские оригиналы.

## Подготовка `.docx`

Подготовка `.docx` — это отдельный этап до JSON-парсинга.

Автоматическая конвертация `.doc` делается через LibreOffice в headless-режиме:

```bash
soffice --headless --convert-to docx --outdir tmp/converted "path/to/file.doc"
```

На macOS бинарь может быть доступен как `soffice`, либо по полному пути:

```bash
/Applications/LibreOffice.app/Contents/MacOS/soffice --headless --convert-to docx --outdir tmp/converted "path/to/file.doc"
```

В текущей рабочей среде LibreOffice уже установлен и доступен двумя способами:

```text
/usr/local/bin/soffice
/Applications/LibreOffice.app/Contents/MacOS/soffice
```

Проверенная версия:

```text
LibreOffice 26.2.3.2
```

Поэтому будущий CLI может сначала вызывать `soffice` из `PATH`, а если команда недоступна, пробовать macOS fallback-путь `/Applications/LibreOffice.app/Contents/MacOS/soffice`.

Будущий CLI подготовки должен делать это автоматически:

1. Найти `.doc` или `.docx` во всех папках `data/source-data/pearls-word/<год>/<квартал>/Брошюры` и `БРОШЮРЫ`.
2. Игнорировать `Оригиналы`, английские исходники и временные Word-файлы `~$...`.
3. Если файл уже `.docx`, скопировать его в `data/word-docx/<год>/<квартал>/Брошюры/`.
4. Если файл `.doc`, создать временную папку вроде `tmp/converted/<hash-or-year-quarter>/`.
5. Вызвать `soffice --headless --convert-to docx --outdir <tmp>` для этого файла.
6. Проверить, что появился `.docx`.
7. Скопировать готовый `.docx` в `data/word-docx/<год>/<квартал>/Брошюры/`.
8. Сохранить структуру по годам и кварталам, чтобы результат можно было смотреть руками.

Конвертация должна быть техническим подготовительным шагом. JSON должен хранить путь к исходному файлу из `data/source-data/pearls-word` и путь к подготовленному `.docx` из `data/word-docx`.

## Полный Pipeline

Целевой поток данных:

```text
data/source-data/pearls-word/
  -> data/word-docx/
  -> data/parsed/
  -> Postgres
```

`data/word-docx/` — промежуточный результат подготовки Word-файлов. `data/parsed/` — reviewed JSON, источник правды для приложения. Postgres — runtime index, который пересобирается из JSON.

Фактический текущий цикл:

```text
npm run prepare:docx
  -> npm run parse:word
  -> npm run db:seed
  -> npm run generate:downloads
  -> npm run build
```

Результат текущего прохода: 54 Word-брошюры и 77 внутренних материалов. Проблемные заголовки и правила разбиения закреплены не ручной правкой JSON, а в `data/word-processing-map.json`.

## Сущности

### Материал

`Материал` остается смысловой единицей внутри жемчужины.

Одна брошюра может содержать один или несколько материалов:

- диктовке;
- лекции;
- проповеди;
- учению;
- молитвенному или другому текстовому материалу.

Материал хранится внутри `documents[]` с собственными `header`, `body`, `footer`, автором, датами и типом.

### Жемчужина

`Жемчужина` снова считается физическим контейнером и основным JSON-документом.

Одна Word-брошюра из папки `Брошюры` соответствует одной жемчужине и одному JSON-файлу в `data/parsed/`.

Внутри жемчужины сохраняется список `documents[]`. Это хорошо совпадает с текущей архитектурой сайта и заменяет старую модель PDF-парсинга.

### Публикационный Блок

Для группировки жемчужин нужна логика публикации:

- год публикации на сайте;
- квартал публикации на сайте;
- месяц публикации на сайте, если известен;
- порядок материала внутри месяца или квартала;
- необязательная метка выпуска.

Эта группировка должна помогать показывать три месячных выпуска внутри квартала, но не должна ломать внутреннюю структуру `documents[]`.

## JSON-Модель

Каждая Word-брошюра парсится в один JSON.

Целевая форма близка к текущей PDF-модели, но источник становится Word:

```json
{
  "slug": "2025-03",
  "title": "Жемчужины Мудрости",
  "sourceWord": "data/source-data/pearls-word/2025/1-й квартал/Брошюры/ЖМ 3_кв. 1_март 2025_....docx",
  "archivePdf": null,
  "sitePublication": {
    "year": 2025,
    "quarter": 1,
    "month": 3,
    "sortDate": "2025-03-01"
  },
  "documentsCount": 3,
  "documents": [
    {
      "documentTitle": null,
      "documentType": "dictation",
      "author": {
        "name": "Архангел Гавриил",
        "slug": "archangel-gabriel",
        "raw": "Диктовка Архангела Гавриила"
      },
      "creation": {
        "date": null,
        "year": null,
        "raw": null
      },
      "pearlPublication": {
        "volume": null,
        "issue": null,
        "date": null,
        "rawDate": null,
        "raw": null
      },
      "parts": {
        "header": [],
        "body": [],
        "footer": []
      }
    }
  ]
}
```

Массив `documents[]` остается основной моделью. Если в брошюре один материал, массив содержит один элемент. Если в брошюре несколько диктовок или лекций, каждая хранится отдельным внутренним документом.

## Ручная Проверка Метаданных

Поскольку Word-брошюры могут отличаться оформлением, парсер не должен пытаться идеально угадать все с первого раза.

Рабочий процесс:

1. Берется один квартал.
2. `.doc` автоматически конвертируется в `.docx`, если нужно.
3. Подготовленные `.docx` сохраняются в `data/word-docx/`.
4. Парсер генерирует JSON из подготовленных `.docx`.
5. Редактор вручную проверяет названия, авторов, даты, типы и разбиение на `documents[]`.
6. Ошибки фиксируются либо правилами парсера, либо редакторскими override-метаданными.
7. После этого JSON сидится в Postgres.
8. После этого обрабатывается следующий квартал или следующий год.

Важно: ручная проверка является частью процесса. Новая цель не в том, чтобы сразу автоматически распарсить весь архив, а в том, чтобы двигаться аккуратно и воспроизводимо.

На текущем этапе ручная проверка дала рабочий набор правил:

- заголовки определяются не только по тексту, но и по форматированию DOCX: bold, italic, font size, style id;
- служебные строки вроде `Жемчужины Мудрости`, `Том ...`, `ПРИЗЫВ`, строк авторов и retreat metadata не должны становиться названием;
- спорные случаи фиксируются в `data/word-processing-map.json`;
- `data/parsed/` по-прежнему не редактируется руками.

## Отношение К Старой PDF-Модели

Старая модель описывала PDF как контейнер, внутри которого могло быть несколько материалов. Для исторических PDF это было правильно.

Новая модель меняет источник, но сохраняет контейнерную идею:

- раньше физический файл мог быть контейнером;
- теперь физический файл тоже остается контейнером, но это Word-брошюра вместо PDF;
- одна брошюра соответствует одному месячному выпуску;
- внутренние материалы по-прежнему живут в `documents[]`.

Старый документ `DOCUMENTS_GUIDE.md` остается полезным для правил заголовка, тела, футера, дат, автора, типа документа и разбиения контейнера на внутренние материалы. Будущий парсер должен применять эти правила к Word-брошюре.

## Что Уже Сделано

1. Архив PDF хранится в `data/source-data/pearls-pdf/`.
2. CLI обходит все папки `data/source-data/pearls-word/<год>/<квартал>/Брошюры` / `БРОШЮРЫ`.
3. `.doc` конвертируется в `.docx` через LibreOffice headless.
4. Подготовленные `.docx` сохраняются в `data/word-docx/<год>/<квартал>/Брошюры/`.
5. Word-first парсер читает подготовленные `.docx`.
6. Парсер читает OpenXML body/header/footer и признаки форматирования.
7. 2022 год и все текущие найденные годы/кварталы прогнаны.
8. `sourceWord` и `preparedDocx` записываются в JSON и БД.
9. `documents[]` сохраняет внутренние материалы брошюры.
10. Seed, каталог и downloads работают поверх Word JSON.

## Принятое Решение

Новая целевая концепция:

```text
одна русская Word-брошюра = одна жемчужина = один JSON
```

Если брошюра хранится в `.doc`, она автоматически конвертируется в `.docx`, и только после этого парсится.

`Жемчужина Мудрости` на сайте остается контейнером публикации, внутри которого могут находиться несколько отдельных материалов. Это ближе к текущей архитектуре проекта и использует уже отработанную модель разбиения на внутренние документы.
