# Миграция Концепции: Word-Оригиналы Вместо PDF

Этот документ фиксирует новую рабочую концепцию архива. Он не удаляет и не заменяет старую модель из `DOCUMENTS_GUIDE.md`, а описывает следующий смысловой поворот перед будущей переработкой парсера.

## Главная Идея

Физическим источником контента становятся отдельные Word-файлы переводов в формате `.docx`.

PDF больше не считается основным источником для парсинга. Папка `pearls-pdf` остается архивом исходных PDF-файлов и может использоваться позже только как справочная ссылка на оригинальный файл, если такая связь нужна на сайте.

Новая модель отказывается от идеи, что один выпуск `Жемчужин Мудрости` обязан быть одним физическим файлом, внутри которого лежит несколько диктовок или лекций. Теперь каждый материал хранится отдельно.

## Источник Данных

Основная папка источников:

```text
pearls-word/
```

Целевая организация источников:

```text
pearls-word/
  2025/
    <отдельный материал>.docx
    <отдельный материал>.docx
  2026/
    <отдельный материал>.docx
```

Промежуточная старая структура с кварталами и папками `Оригиналы` может использоваться во время ручного переноса:

```text
pearls-word/2025/1-й квартал/Оригиналы/
```

Но целевая идея проще: складывать только русские `.docx`-оригиналы переводов сразу по годам, без лишней вложенности.

## Что Парсим

Парсер должен обрабатывать только оригинальные русские `.docx`-файлы переводов.

На первом этапе не парсим:

- PDF-файлы из `pearls-pdf`;
- скомпонованные месячные брошюры;
- английские оригиналы;
- временные Word-файлы вида `~$...`;
- старые `.doc`, если они не были заранее конвертированы в `.docx`.

Если позже понадобится поддержка `.doc`, ее лучше добавлять отдельным шагом через конвертацию в `.docx`, а не смешивать с основной логикой.

## Сущности

### Материал

`Материал` становится главной физической и смысловой сущностью.

Один `.docx`-файл должен соответствовать одному материалу:

- диктовке;
- лекции;
- проповеди;
- учению;
- молитвенному или другому текстовому материалу.

Именно материал получает собственный JSON-файл в `data/parsed/`.

### Жемчужина

`Жемчужина` больше не является обязательным физическим файлом.

В новой модели это логическая публикационная группировка: набор отдельных материалов, которые вышли на сайте в одном месяце, квартале или выпуске.

То есть материалы не склеиваются насильно в один файл. Они остаются отдельными JSON-документами, а сайт может показывать их рядом как один логический блок публикации.

### Публикационный Блок

Для группировки материалов нужна отдельная логика публикации:

- год публикации на сайте;
- квартал публикации на сайте;
- месяц публикации на сайте, если известен;
- порядок материала внутри месяца или квартала;
- необязательная метка выпуска.

Эта группировка должна быть индексом или метаданными поверх отдельных материалов, а не причиной объединять тексты в один физический документ.

## JSON-Модель

Каждый `.docx`-материал парсится сразу в свой JSON.

Целевая форма может остаться близкой к текущей, но смысл меняется:

```json
{
  "slug": "2025-03-archangel-gabriel",
  "title": "Диктовка Архангела Гавриила",
  "sourceWord": "pearls-word/2025/...",
  "archivePdf": null,
  "sitePublication": {
    "year": 2025,
    "quarter": 1,
    "month": 3,
    "sortDate": "2025-03-01"
  },
  "document": {
    "documentTitle": null,
    "documentType": "dictation",
    "author": {
      "name": "Архангел Гавриил",
      "slug": "archangel-gabriel",
      "raw": "Диктовка Архангела Гавриила"
    },
    "creation": {
      "date": null,
      "year": null,
      "raw": null
    },
    "pearlPublication": {
      "volume": null,
      "issue": null,
      "date": null,
      "rawDate": null,
      "raw": null
    },
    "parts": {
      "header": [],
      "body": [],
      "footer": []
    }
  }
}
```

На практике можно временно сохранить текущий массив `documents[]`, но в новой концепции он почти всегда будет содержать один элемент. После стабилизации можно решить, упрощать ли модель до одного `document`.

## Ручная Проверка Метаданных

Поскольку Word-оригиналы оформлены неодинаково, парсер не должен пытаться идеально угадать все с первого раза.

Рабочий процесс:

1. Берется небольшой квартал или год.
2. Парсер генерирует JSON из `.docx`.
3. Редактор вручную проверяет названия, авторов, даты и типы.
4. Ошибки фиксируются либо правилами парсера, либо редакторскими override-метаданными.
5. После этого обрабатывается следующий квартал.

Важно: ручная проверка является частью процесса. Новая цель не в том, чтобы сразу автоматически распарсить весь архив, а в том, чтобы двигаться аккуратно и воспроизводимо.

## Отношение К Старой PDF-Модели

Старая модель описывала PDF как контейнер, внутри которого могло быть несколько материалов. Для исторических PDF это было правильно.

Новая модель меняет источник:

- раньше физический файл мог быть контейнером;
- теперь физический файл должен быть отдельным материалом;
- контейнер публикации становится логической сущностью сайта;
- группировка материалов делается через метаданные, а не через склейку текстов.

Старый документ `DOCUMENTS_GUIDE.md` остается полезным для правил заголовка, тела, футера, дат, автора и типа документа. Но будущий парсер должен применять эти правила к одному Word-материалу, а не к PDF-контейнеру.

## Будущий План Переработки

Когда начнется реализация, изменения стоит делать отдельными шагами:

1. Добавить Word-first парсер для `.docx`.
2. Научить CLI читать `pearls-word`, а не `pearls` или `pearls-pdf`.
3. Переименовать PDF-зависимые поля в нейтральные: `sourcePdf` -> `sourceWord` или `sourcePath`.
4. Убрать из новой генерации PDF-специфику: страницы, колонки, layout, `pdf-processing-map`.
5. Добавить простую модель публикационных блоков для группировки отдельных материалов на сайте.
6. Обновить сидинг базы и каталог под материалы как основные единицы.
7. Только после этого решать, нужна ли связь с архивными PDF.

## Принятое Решение

Новая целевая концепция:

```text
один русский .docx оригинал перевода = один материал = один JSON
```

`Жемчужина Мудрости` на сайте становится логическим блоком публикации, который объединяет отдельные материалы по редакционной дате, но не заставляет хранить их в одном файле.
